# Airdrop Dashboard Manager App Flow

## Onboarding and Sign-In/Sign-Up
When a brand-new visitor arrives at the application by entering its URL in a browser or clicking a shared link, they first land on the sign-in page. The page clearly invites them to either sign in or create a new account. To sign up, the user clicks the link to the sign-up form, types in their email address and a password, and submits the form. The application uses an API route under `/api/auth/signup` to register the new user and then redirects them to the sign-in page. Returning users simply provide their email and password on the sign-in page. On form submission, the credentials go to the `/api/auth/signin` endpoint for verification. If the login succeeds, the system sets a secure session cookie and automatically navigates the user to the main dashboard. If they ever need to leave, a prominent “Sign Out” button in the header calls the `/api/auth/signout` endpoint, clearing the session and returning them to the sign-in screen.

## Main Dashboard or Home Page
Once authenticated, the user lands on the dashboard home screen. At the top, a fixed header displays the application logo, the title “Airdrop Dashboard,” and the “Sign Out” button. Along the left edge, a slim navigation bar highlights the active “Dashboard” link. No other menu items appear at this stage. The central area of the page shows a collection of visual cards or tables, each representing an active airdrop campaign. These cards list the campaign name, its current status, the total number of participants, and progress bars showing distribution metrics. At the bottom of the page, a simple footer notes the application version and copyright.

## Detailed Feature Flows and Page Transitions
When the dashboard first renders, the page component triggers a data fetch. It calls a backend API or loads the local `data.json` file, then enters a loading state where a spinner or placeholder appears in place of the cards. When the data arrives, each campaign card populates with real numbers and status labels. If the user wants fresh data, they click a refresh icon in the header area, which repeats the fetch and briefly shows the spinner again. In the event that the data set is empty, the page displays a friendly message explaining that no campaigns are available and prompts the user to check back later.

If the user clicks on one of the campaign cards, the interface smoothly expands that card into a detailed view section. This in-place detail view lists participant breakdowns, distribution logs, and the date of the last update. A “Back to Dashboard” link at the top of the detail panel closes it and returns the user to the full set of cards. Throughout these transitions, the left navigation bar and header remain visible, ensuring the user always knows where they are in the app flow.

## Settings and Account Management
Currently, personalized account settings are minimal. The only direct account action is signing out, which appears in the header at all times. Since there is no dedicated settings page, users cannot yet update their email, change their password, or configure notification preferences within the application. In future versions, a new “Settings” link could appear in the navigation bar, guiding users to a form where they manage personal details. After any settings change, the user would click a “Save” button and the app would confirm the update before returning to the dashboard view.

## Error States and Alternate Paths
If a user enters the wrong email or password on the sign-in form, an inline error message appears below the input fields explaining that the credentials are invalid. The form remains on screen so they can try again. During dashboard data fetching, network problems or server errors display a colored banner across the top of the main content area. This banner explains that data could not be loaded and offers a “Retry” button. If the user’s session cookie expires or becomes invalid, any attempt to fetch data will redirect them automatically back to the sign-in page with a brief alert that their session has ended.

## Conclusion and Overall App Journey
In a typical user journey, someone new visits the app, signs up for an account, and then signs in with their newly created credentials. They arrive at the dashboard, where they review real-time metrics for their airdrop campaigns. They may expand individual campaign cards to see detailed breakdowns or hit the refresh icon to update statistics. When they finish, they click the sign-out button and return to the sign-in page. Throughout every step, the flow remains clear, with persistent navigation elements and immediate feedback on all actions, ensuring that users can manage and monitor their airdrops with confidence and ease.
